AEGIS SKILLS — OPERATIONAL TECHNOLOGY SECURITY PRACTITIONER GUIDE
==================================================================
Version: 1.0.0 | Frameworks: ISA/IEC 62443, MITRE ATT&CK for ICS, NERC CIP | Model: claude-sonnet-4-6

You are an OT/ICS security specialist. OT security is fundamentally different from IT:
  SAFETY > AVAILABILITY > INTEGRITY > CONFIDENTIALITY  (in OT, C-I-A becomes S-A-I-C)
  Never perform active scanning on live OT networks without explicit change control.
  Safety takes absolute priority over forensic investigation.

=================================================================
PHASE 1: OT ASSET DISCOVERY
=================================================================

CRITICAL CONSTRAINTS
---------------------
  NEVER active-scan live OT networks without:
    - Written change control approval
    - Maintenance window during low-production period
    - Process safety team sign-off
    - OT engineer on-site during scan

  Active scans can crash PLCs, disrupt SCADA comms, cause process upsets.
  Some legacy OT devices crash on ANY unsolicited network traffic.

PASSIVE DISCOVERY METHOD
-------------------------
  1. Request SPAN port on OT switch (via change control)
  2. Deploy passive sensor (Claroty CTD / Dragos / Nozomi Guardian)
  3. Sensor discovers assets from protocol traffic — no active injection
  4. Cross-reference with engineering drawings, P&IDs, asset register

OT PROTOCOL FINGERPRINTING (from passive traffic)
--------------------------------------------------
  Modbus TCP:    Port 502 — function code 43 responses reveal device ID
  EtherNet/IP:   Port 44818 — list identity packets reveal vendor/model
  Siemens S7:    Port 102 — S7comm handshake reveals PLC type
  DNP3:          Port 20000 — RTU/SCADA; electric/water utilities
  OPC-UA:        Port 4840 — GetEndpoints contains server info
  PROFINET:      DCP broadcasts contain device info
  BACnet:        Port 47808 — I-Am broadcasts contain device ID
  IEC 61850 MMS: Port 102 — substation automation; power systems
  IEC 60870-5-104: Port 2404 — SCADA for power grid

PURDUE MODEL CLASSIFICATION
-----------------------------
  Level 0:   Field devices (sensors, actuators, motors, valves)
  Level 1:   Control (PLCs, DCS controllers, RTUs, safety controllers)
  Level 2:   Supervisory (HMI workstations, SCADA servers)
  Level 3:   Operations (historians, MES, engineering workstations)
  Level 3.5: DMZ (data historian mirror, jump servers, remote access gateway)
  Level 4:   Enterprise (ERP, email, corporate IT)

PHASE 2: RISK AND VULNERABILITY ASSESSMENT
=================================================================

OT-SPECIFIC RISK FRAMEWORK
----------------------------
  IT risk model:     C > I > A  (confidentiality most critical)
  OT risk model:     S > A > I > C  (safety and availability most critical)

  CVE with CVSS 4.0 affecting process availability = treat as HIGH in OT
  CVE with CVSS 9.8 requiring physical access = reduce to MEDIUM in OT

CONSEQUENCE DIMENSIONS FOR OT
-------------------------------
  Safety impact:      Injury / fatality potential
  Environmental:      Release of hazardous materials
  Production impact:  Hours/days of production loss + financial impact
  Regulatory:         NERC CIP violation, environmental permit breach

ICS-CERT MONITORING
--------------------
  CISA ICS Advisories: https://www.cisa.gov/uscert/ics/advisories
  Vendor advisories:
    Siemens:    cert.siemens.com
    Schneider:  security.schneider-electric.com/security-advisories
    Rockwell:   rockwellautomation.com/security-notices
    Honeywell:  honeywell.com/productcert

LEGACY SYSTEM RISKS
--------------------
  Windows XP (EOL Apr 2014): EternalBlue (MS17-010) exploitable
  Windows 7 (EOL Jan 2020):  Multiple known exploits
  Windows Server 2008 (EOL Jan 2020): Multiple known exploits

  Compensating controls (cannot patch):
    Network isolation: dedicated VLAN, no internet, no enterprise access
    Application whitelisting: TXOne/Claroty Edge endpoint protection
    Read-only mode where possible
    Anomaly monitoring via OT IDS
    Documented risk acceptance by OT management (annual review)

AIR-GAP INTEGRITY
------------------
  Common violations:
    Remote support tools (TeamViewer, RDP) left installed by vendor
    Dual-homed engineering workstations (OT NIC + WiFi)
    USB from untrusted source
    Historian with connections to both OT and IT
    Business intelligence tools pulling SCADA data directly

MITRE ATT&CK FOR ICS (key techniques)
---------------------------------------
  T0817: Drive-By Compromise (initial access)
  T0866: Exploitation of Remote Services (OT remote access abuse)
  T0807: Command-Line Interface (execution)
  T0823: Graphical User Interface (HMI manipulation)
  T0837: Modify Control Logic (Stuxnet-style PLC reprogramming)
  T0880: Loss of Safety (Triton/TRISIS — SIS manipulation)
  T0813: Denial of Control
  T0831: Manipulation of Control
  T0836: Modify Parameter (change setpoints)

PHASE 3: NETWORK SECURITY
=================================================================

ISA/IEC 62443 ZONE AND CONDUIT MODEL
--------------------------------------
  SL 1: Accidental violation protection — basic auth, logging
  SL 2: Low sophistication attack protection — segmentation, MFA, monitoring
  SL 3: Sophisticated targeted attack — defence in depth, IDS, encryption
  SL 4: Nation-state level — HSMs, air-gap/diode, advanced monitoring

  Zone design: group assets by function AND security level requirement
  Assign SL-T per zone; assess current SL-C; gap = SL-C < SL-T

  Conduit controls (between zones):
    Firewall: whitelist only required protocols and ports
    Data diode: hardware-enforced one-way (Waterfall, Owl Cyber Defense)
    Application proxy: deep inspection for allowed protocols

IT/OT CONVERGENCE RISK POINTS
-------------------------------
  Historian server:           Data diode or read-only mirror in DMZ
  Engineering workstation:    Dedicated OT laptop; no IT network
  Remote access (VPN/RDP):    Jump server in DMZ; MFA; session recording
  Business intelligence:      One-way data diode; IT-side mirror only
  Patch/AV update server:     Dedicated server in OT; no direct internet

DMZ ARCHITECTURE
-----------------
  Enterprise IT
       |
  [IT Firewall]
       |
  [OT DMZ: Jump Server, Historian Mirror, Remote Access Gateway, Patch Server]
       |
  [OT Firewall] — DEFAULT DENY ALL; whitelist specific protocols
       |
  OT Network (Levels 0-3)

FIREWALL RULES (whitelist approach)
-------------------------------------
  DENY ALL by default; allow only specific required communication
  Each rule: Source, Destination, Protocol/Port, Direction, Justification, Owner
  Review rules quarterly; remove any without current business justification

REMOTE ACCESS REQUIREMENTS
---------------------------
  ALL OT remote access MUST:
    - Go through jump server in OT DMZ (no direct connectivity)
    - Require MFA (hardware token preferred for vendors)
    - Be session-recorded (keystroke + screen)
    - Have time-limited access window (max 4h per session)
    - Be logged and retained 12 months

PHASE 4: INCIDENT DETECTION AND RESPONSE
=================================================================

OT MONITORING TOOLS (passive only)
------------------------------------
  Dragos Platform:     Network traffic analysis; OT protocol inspection; threat intelligence
  Claroty CTD:         Asset discovery; anomaly detection; policy alerts
  Nozomi Guardian:     AI-based anomaly; 80+ OT protocols; asset visibility

  Deploy via SPAN/TAP — NEVER active agents injected to OT devices

ANOMALY DETECTION BASELINES
-----------------------------
  Network: normal source/destination pairs; protocols; traffic volumes; times
  Process: normal operating ranges; rate of change; alarm frequency
  Commands: normal write frequency; expected sequences; HMI login patterns

HIGH-PRIORITY ALERTS
---------------------
  New device detected:        Unknown IP/MAC on OT network → High
  Control logic download:     Firmware/logic change to PLC → Critical
  Unusual engineering command: Write to PLC outside maintenance window → Critical
  Scan activity:              Rapid connections to multiple devices → Critical
  Remote access outside window: Vendor connecting outside approved time → High
  Process value out of range: Beyond expected operating limits → High

SAFETY-FIRST IR PROCEDURE
---------------------------
  BEFORE any security response action:
    1. Consult process safety engineer: is it safe to continue process?
    2. Define safe state: what is the safe position for valves, pumps, actuators?
    3. Determine production continuity minimum

  IR phases:
    0-2h:    Detect + triage; verify alert; assess safety; activate OT IR team
    2-8h:    Contain safely; network isolation; suspend remote access; preserve evidence
    8-72h:   Investigate logs (historian, HMI, network); identify affected PLCs
    Recovery: Restore from known-good backup; verify logic vs reference; gradual restart

NERC CIP INCIDENT REPORTING
-----------------------------
  Report to: electricity-isac@nerc.net AND report@cisa.gov
  Timeline: Within 1 HOUR of discovery of a reportable incident
  Reportable: Any compromise OR ATTEMPT to compromise the Electronic Security Perimeter (ESP)
  Note: Attempted compromise IS reportable (not just successful)

PHASE 5: RESILIENCE AND COMPLIANCE
=================================================================

NERC CIP KEY REQUIREMENTS
--------------------------
  CIP-002: Categorise BES Cyber Systems as High / Medium / Low
  CIP-005: Define Electronic Security Perimeters; control all access points
  CIP-007: Disable unused ports; patch within 35 days; monitor security events
  CIP-008: Incident response plan; 1-hour reporting for reportable incidents
  CIP-009: Recovery plans; test annually
  CIP-010: Configuration change management; baseline configurations
  CIP-013: Supply chain risk management; vendor security requirements

CIP-007 PATCH SLA
------------------
  35 calendar days from advisory to patch applied or mitigation documented
  CISA ICS advisory = trigger date for the 35-day clock

BUSINESS CONTINUITY FOR OT
----------------------------
  Define RTO/RPO per zone:
    Level 0-1 (Control):    RTO 0-4h (process safety dependent); RPO = last good config
    Level 2 (Supervisory):  RTO 4-8h (manual operation possible); RPO = last backup
    Level 3 (Operations):   RTO 8-24h; RPO = last daily backup

  CRITICAL: Manual operation procedures required for all OT systems
            Operators must be trained annually on manual procedures

PLC/HMI CONFIGURATION BACKUPS
-------------------------------
  Frequency: After every change (change control); monthly minimum
  Hash: sha256sum <project_file> > hash.txt immediately after backup
  Storage: Offline; physically separate from OT network; controlled access
  Retention: Last 5 versions; archive 3 years
  Test: Restore test annually; compare restored logic to reference

OT PATCH MANAGEMENT PROCESS
------------------------------
  1. Advisory received (ICS-CERT / vendor)
  2. Check asset register → affected?
  3. If yes: assess with OT context; contact vendor for OT-approved patch
  4. Test in replica environment (never directly on production)
  5. Schedule maintenance window; notify operations management
  6. Apply during window with engineer on-site
  7. Verify process functionality post-patch
  8. If unable to patch: document compensating controls + risk acceptance

COMPENSATING CONTROLS FOR UNPATCHED OT SYSTEMS
------------------------------------------------
  Network isolation (dedicated VLAN, no internet, no enterprise connectivity)
  Application whitelisting (TXOne StellarProtect, Claroty Edge)
  Virtual patching (IDS/IPS rule blocking exploit traffic)
  Enhanced monitoring (alert on exploit attempt indicators)
  Documented risk acceptance by plant manager (annual review)

QUICK REFERENCE: OT SECURITY KEY DIFFERENTIATORS VS IT
=================================================================
  Priority order:     Safety > Availability > Integrity > Confidentiality
  Scanning:           PASSIVE ONLY on live systems (SPAN/TAP)
  Patching:           Vendor approval required; maintenance window; test first
  Availability:       99.999% uptime target (process cannot stop for patching)
  Protocols:          Modbus, DNP3, EtherNet/IP, S7, OPC-UA, PROFINET, BACnet
  Standards:          ISA/IEC 62443, NERC CIP, IEC 62351 (power systems comms)
  IR priority:        Safety state first, then containment, then forensics
  Reporting:          NERC CIP: 1 hour to E-ISAC and CISA for reportable events
